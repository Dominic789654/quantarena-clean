## Context

Seven prior Phase 3 changes moved `BacktestDecision`, `SharedPhase1Artifact`/`SharedPhase1ArtifactCache`, `SharedAnalystSignalCache`, the scoring functions, the decision-apply helpers, the db-store glue, the company-news-signature resolver, the signal-collection engine, and the phase1 pipeline out of `backtest/workflow_adapter.py` into `backtest/workflow/`. What remains (832 lines) is `BacktestWorkflowAdapter`'s own core: `__init__`/config resolution, `_adopt_prev_portfolio`, `_build_api_source_config`, the three public `run_single_day*` orchestration methods (which call the AgentWorkflow/graph machinery directly — never extracted, and out of scope here), `get_current_portfolio`, `close`/`__enter__`/`__exit__`/`__del__`, roughly twenty same-named delegator methods to the seven already-extracted modules, and the module-level `create_workflow_adapter` factory.

## Goals / Non-Goals

**Goals:** relocate the class and factory into `backtest/workflow/adapter.py` (completing the package migration — every other Phase 3 piece already lives under `backtest/workflow/`), verbatim; collapse `backtest/workflow_adapter.py` into a minimal, clearly-documented compatibility shim; prove every existing import path and monkeypatch string in the codebase and test suite still resolves to the same objects after the move.

**Non-Goals:** touching `run_single_day`/`run_single_day_with_precollected_signals`/`run_single_day_with_smart_priority`'s bodies (already characterization-tested in an earlier Phase 3 step; this step moves them unchanged); touching any of the seven previously-extracted modules; the optional deferred step 23 (`migrate-workflow-adapter-direct-consumers`, updating `backtest/engine.py`/`backtest/multi_personality_engine.py`/tests to import from `backtest.workflow.adapter` directly) — out of scope, the shim exists precisely so that migration can happen later, independently, or never.

## Decisions

1. **Re-export shim, no `sys.modules` indirection.** The run.py track's `runner/_shim.py` pattern (a `sys.modules.get('run') or sys.modules.get('__main__')` lookup) exists to defend against `monkeypatch.setattr("run.<fn>", fake)` where a caller *inside the moved code* resolves `<fn>` through its own module's global namespace (`fn()`, not `self.fn()` or `module.fn()`) — a plain re-export breaks that case because the caller's globals still point at the old, un-patched function object. This class of hazard does not exist here: every one of the three monkeypatch targets found in this audit (`SharedPhase1ArtifactCache.ARTIFACT_VERSION`, `BacktestWorkflowAdapter.SHARED_PHASE1_ARTIFACT_VERSION`, `logger.info`) is a *class-attribute or singleton-object attribute* patch — the target string's job is only to *locate* the object (via `getattr(module, name)` chains), and once located, the patch mutates the object itself, which is then observed by every reference to that same object regardless of which module holds it. A re-export makes the shim's `getattr(module, name)` chain succeed; it does nothing else, and nothing else is needed.
2. **The shim keeps a bare `from loguru import logger` import**, not merely relying on it being reachable transitively through `backtest.workflow.adapter`. This is the one re-export in the shim that exists for a monkeypatch, not an import-path reason (no code imports `logger` from `backtest.workflow_adapter`) — `monkeypatch.setattr('backtest.workflow_adapter.logger.info', ...)` requires `backtest.workflow_adapter` to have a `logger` *attribute* to navigate through before reaching `.info`. Since `loguru.logger` is a true process-wide singleton (one object per interpreter, not per-import-site), the shim's own `from loguru import logger` and `backtest/workflow/adapter.py`'s own `from loguru import logger` bind the identical object — patching `.info` through either module path has the identical effect. Verified experimentally by temporarily substituting a decoy object for the shim's `logger` (see proposal.md's Impact section): the patch then silently stopped taking effect, confirming the identity requirement and ruling out reliance on accidental attribute passthrough.
3. **`backtest/workflow/adapter.py` gets its own full copy of every import the class needs**, including the module-level `setup_paths()`/`load_dotenv(...)` calls, rather than the shim performing setup and adapter.py assuming it already happened. This makes `backtest/workflow/adapter.py` independently importable (e.g. `from backtest.workflow.adapter import BacktestWorkflowAdapter` without going through the shim at all) with identical behavior to importing through `backtest.workflow_adapter` — consistent with every prior Phase 3 extraction's independence, and required by ground rule 2 (verbatim move: the original module's side effects were tied to importing *a* module containing this class; that must remain true for whichever module now contains it, not conditionally require the shim).
4. **Full suite run twice.** The instructions call this out explicitly because this step moves the methods that orchestrate the threaded signal-collection engine (`run_single_day_with_smart_priority` -> `collect_signals_only_parallel_v2` -> `ThreadPoolExecutor`) into a new module; although the thread-pool code itself was already extracted and audited in the signal-collection-engine step, the *orchestration* call sites move here, so the same nondeterminism precaution applies.

## Risks / Trade-offs

- This is a large, mechanical move (the bulk of the remaining file) but the lowest-*novel*-risk step in the program: every hazard category was already identified and closed by earlier Phase 3 steps' discipline (adapter-passing, same-named delegators, singleton-attribute patch awareness). The only genuinely new risk surface is the shim's re-export completeness, closed by the two `git grep` audits in tasks.md and the experimental `logger` identity check.
- Leaving `backtest/engine.py`/`backtest/multi_personality_engine.py`/every test file importing from the old `backtest.workflow_adapter` path (rather than migrating them to `backtest.workflow.adapter`) is intentional per ground rule 4 (deferred, optional final step) — changing them is not required for correctness and would inflate this change's diff without reducing risk.
