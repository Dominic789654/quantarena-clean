## Why

Phase 3 step 22 (docs/refactor_program_plan.md) — the finale of the workflow_adapter decomposition program. After the previous seven extractions, `backtest/workflow_adapter.py` (832 lines) contains nothing but `BacktestWorkflowAdapter`'s core (init/config, `_adopt_prev_portfolio`, `_build_api_source_config`, the three `run_single_day*` orchestration methods, `get_current_portfolio`, `close`/context-manager, the ~20 same-named delegator methods to the seven already-extracted modules) and the `create_workflow_adapter` factory, plus a handful of re-imports kept for backward compatibility. This step moves that remaining core into `backtest/workflow/adapter.py` (matching the `backtest/workflow/` package every other piece already lives in) and collapses `backtest/workflow_adapter.py` into a thin compatibility shim.

`git grep -n "backtest.workflow_adapter" -- backtest/ tests/ runner/ deepfund/` finds three categories of external reference, all satisfied by re-export rather than needing any `sys.modules` indirection: (a) plain imports (`from backtest.workflow_adapter import BacktestWorkflowAdapter`, `BacktestDecision`, `create_workflow_adapter`, `SharedPhase1Artifact` — in `backtest/engine.py`, `backtest/multi_personality_engine.py`, and six test files) — resolved by re-export; (b) two class-attribute monkeypatches, `monkeypatch.setattr("backtest.workflow_adapter.SharedPhase1ArtifactCache.ARTIFACT_VERSION", "v3")` and `monkeypatch.setattr("backtest.workflow_adapter.BacktestWorkflowAdapter.SHARED_PHASE1_ARTIFACT_VERSION", "v3")` (`tests/test_multi_personality_day_orchestrator.py:694-695`) — both resolve `backtest.workflow_adapter.<Name>` to a *class object* and mutate an attribute on that object directly, so they are unaffected by which module defines the class, as long as the shim re-exports the same object; (c) one attribute-of-a-singleton patch, `monkeypatch.setattr('backtest.workflow_adapter.logger.info', ...)` (`tests/test_workflow_adapter_smart_priority.py:340`) — `loguru.logger` is a process-wide singleton, so patching `.info` on the object reached via `backtest.workflow_adapter.logger` mutates the exact same object that `backtest/workflow/adapter.py`'s own `from loguru import logger` binds; this requires the shim to keep exposing `logger` as a module attribute (unlike the run.py track's bare-function-rebinding hazard, this is not a module-global-rebinding case — no internal caller in `backtest/workflow/adapter.py` looks up `logger` through the shim module, it uses its own `from loguru import logger` import, which is the identical singleton object).

No case in this audit needs the `backtest/workflow/_compat.py` / `sys.modules['backtest.workflow_adapter']` indirection pattern used in the run.py track: that pattern defends against `monkeypatch.setattr("module.bare_function", fake)` where the *caller* resolves `bare_function` through its own module globals (a name rebinding). Every risky pattern found here is either a plain re-export (module attribute lookup, survives moving the definition) or a mutation of a shared object's own attribute (survives regardless of which module holds a reference to that object). This was verified experimentally (see Impact).

## Impact

- New `backtest/workflow/adapter.py` (the full class + factory, ~800 lines). `backtest/workflow_adapter.py` shrinks to a shim (docstring + re-exports, under 25 lines).
- Import-path audit (ground rule 4): `git grep -n "from backtest.workflow_adapter import\|from backtest import" tests/ backtest/ runner/` — every import (`BacktestWorkflowAdapter`, `BacktestDecision`, `SharedPhase1Artifact`, `create_workflow_adapter`) is satisfied by the shim's re-exports; `backtest/__init__.py`'s lazy `_EXPORTS` table still points at `("backtest.workflow_adapter", "BacktestWorkflowAdapter")` etc. — `importlib.import_module("backtest.workflow_adapter")` + `getattr(module, "BacktestWorkflowAdapter")` resolves through the shim's re-export exactly as it resolved through the original definition.
- Monkeypatch audit (ground rule 3), verified experimentally: after landing the move, ran `tests/test_multi_personality_day_orchestrator.py::test_shared_phase1_artifact_cache_key_or_version_change_invalidates` (exercises both class-attribute patches) and `tests/test_workflow_adapter_smart_priority.py` (exercises the `logger.info` patch) standalone — both green. Additionally, as a targeted regression probe, temporarily changed `backtest/workflow_adapter.py`'s `from loguru import logger` line to `import logging as logger` (a different object) and reran `tests/test_workflow_adapter_smart_priority.py`'s `logger.info`-patching test: it failed (the patched `.info` was never called, since `backtest/workflow/adapter.py`'s real logging calls use the genuine `loguru.logger` singleton, untouched by patching the decoy). Reverted before landing — this confirms the shim's `logger` re-export must be the *same* `loguru.logger` object, which the plain `from loguru import logger` import satisfies (loguru guarantees one process-wide singleton per interpreter), and that no `sys.modules` indirection is needed for this case.
- `.venv_unified/bin/python -m pytest tests/ -q` run **twice**, back to back, since this step moves threaded orchestration (`run_single_day_with_smart_priority`'s downstream calls into the signal-collection engine) into a new module: both runs 945 passed, 10 skipped, 0 failed.
- `.venv_unified/bin/ruff check .` clean; `.venv_unified/bin/python run.py --check-env` exits 0.
- Final `backtest/workflow_adapter.py` line count and both suite-run tails recorded in the commit message.
