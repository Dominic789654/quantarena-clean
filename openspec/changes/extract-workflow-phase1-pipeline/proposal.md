## Why

Phase 3 step 21 (docs/refactor_program_plan.md) ā€” the cache-aside composition that sits directly on top of the signal-collection engine extracted in step 20. `_build_shared_phase1_artifact` and `load_or_compute_shared_phase1` (currently `backtest/workflow_adapter.py` lines 624-740) are the day-level shared phase1 artifact pipeline: `load_or_compute_shared_phase1` resolves an input signature via the company-news-signature delegators, attempts a `SharedPhase1ArtifactCache.load`, and on a miss falls back to `collect_signals_only_parallel_v2` before building a fresh `SharedPhase1Artifact` via `_build_shared_phase1_artifact` and writing it back through `SharedPhase1ArtifactCache.save`. `backtest/multi_personality_engine.py`'s `_load_shared_phase1_artifact` calls `shared_signal_adapter.load_or_compute_shared_phase1(trading_date=..., prices=..., max_workers=...)` on real (and, in `tests/test_multi_personality_day_orchestrator.py`, subclassed) adapter instances; several tests in that file define subclasses that override `load_or_compute_shared_phase1` and call `super().load_or_compute_shared_phase1(...)`, so the name must stay a real method resolvable through the MRO, not merely an instance attribute. `tests/test_multi_personality_day_orchestrator.py` also patches `backtest.workflow_adapter.SharedPhase1ArtifactCache.ARTIFACT_VERSION` and `backtest.workflow_adapter.BacktestWorkflowAdapter.SHARED_PHASE1_ARTIFACT_VERSION` as class attributes ā€” both resolve through the shared class objects (re-exported, not redefined), so they are unaffected by which module defines the two functions being moved here.
