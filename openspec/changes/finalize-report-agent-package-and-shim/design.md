## Context

Seven prior Phase 4 changes moved `run_agent_with_retry`, `sanitize_json_chart_blocks`/`build_structured_report`, the six citation-manager functions, `clean_ticker`/`signal_mentions_ticker`/`extract_forecast_requests`/`build_forecast_map`, `process_charts`, and `cluster_signals` out of `deepear/src/agents/report_agent.py` into `deepear/src/agents/report/`. What remains (731 lines) is `ReportAgent`'s own core: `__init__` (constructs the RAG store and the four internal `Agent`s: planner, writer, editor, section_editor), `_get_forecast_agent` (the lazy Kronos-backed cache), `generate_report` (the Map-Reduce orchestration), `_incremental_edit`, and roughly fifteen same-named delegator methods to the seven already-extracted modules. Every one of those prior steps explicitly deferred building out `deepear/src/agents/report/__init__.py`'s re-exports to this step (see each step's design.md "Non-Goals": "building out `deepear/src/agents/report/__init__.py` re-exports (deferred to `finalize-report-agent-package-and-shim`, step 31)").

## Goals / Non-Goals

**Goals:** relocate `ReportAgent` into `deepear/src/agents/report/agent.py`, verbatim; collapse `deepear/src/agents/report_agent.py` into a minimal, clearly-documented compatibility shim; build out `deepear/src/agents/report/__init__.py` into a real package `__init__.py` re-exporting `ReportAgent` and every leaf module's public function; prove every existing import path still resolves; **honestly repair** every monkeypatch that targeted the old module's namespace, by moving the patch to the module where `ReportAgent`'s methods now actually resolve those names, rather than leaving tests that silently stop testing what they claim to.

**Non-Goals:** changing `ReportAgent`'s behavior in any way; touching any of the eight already-extracted leaf modules; the optional deferred step 32 (`migrate-report-agent-direct-consumers`, updating `deepear/src/agents/__init__.py`/`deepear/src/main_flow.py` to import from `deepear.src.agents.report.agent` directly) â€” out of scope, the shim exists precisely so that migration can happen later, independently, or never; adding a `sys.modules` indirection layer (see Decision 2 for why this case does not need one).

## Decisions

1. **Every `Agent`/`ForecastAgent`/`_cluster_signals_impl` monkeypatch site is moved to `deepear.src.agents.report.agent`, not left pointed at the shim.** This is the one place this step deviates from a pure "re-export and done" shim (contrast `backtest/workflow_adapter.py`'s finale, which needed zero test changes because every patch there targeted a *shared object's own attribute* â€” a class attribute or the `loguru.logger` singleton â€” not a plain module-global name). Here, `Agent`, `ForecastAgent`, and `_cluster_signals_impl` are ordinary module-level names that `ReportAgent`'s methods read via bare identifier lookup (`Agent(...)`, `ForecastAgent(self.db, self.model)`, `_cluster_signals_impl(signals, user_query, planner=self.planner)`), resolved against whichever module's `__dict__` the *method's code object* was compiled into â€” that is now `deepear.src.agents.report.agent`, permanently, regardless of which module re-exports the name afterward. Re-exporting `Agent` from the shim (as `from deepear.src.agents.report.agent import Agent` in `report_agent.py`) would make `deepear.src.agents.report_agent.Agent` resolve to *something*, but patching it would mutate the shim's own binding, not the one `ReportAgent.__init__` actually reads â€” a purely cosmetic fix that leaves the test's patch silently inert. The honest fix is to patch where the lookup actually happens.
2. **No `sys.modules` indirection layer** (unlike the run.py track's `runner/_shim.py`, which exists for a different, narrower hazard: a caller *inside a moved function* reading a bare name through its own module globals, where the caller function itself could be independently re-imported and called both via the old and new path). That hazard doesn't apply here â€” the only callers of `Agent`/`ForecastAgent`/`_cluster_signals_impl` are `ReportAgent`'s own methods, which live in exactly one place (`report/agent.py`) after this move, with no parallel code path re-executing the same logic against a different module's globals. A five-file, five-line test edit is simpler, more honest about what changed, and exactly matches the two precedent finales' "no case needs `sys.modules` indirection" findings once the specific hazard is absent.
3. **`tests/report_agent_harness.py`'s edit is one line** (`import deepear.src.agents.report_agent as report_agent_module` â†’ `import deepear.src.agents.report.agent as report_agent_module`), because every `monkeypatch.setattr(report_agent_module, "Agent"/"ForecastAgent", ...)` call site already goes through the `report_agent_module` variable, never a literal string â€” renaming the import target is sufficient and every one of the ~50 tests that call `make_report_agent` picks up the fix for free, which is why the harness was built as a single centralized construction point in the first place (`build-report-agent-characterization-harness`'s design.md goal).
4. **The two string-literal patches** (`test_report_agent_characterization.py`, `test_report_agent_citations.py`) and **the two chart-renderer `import ... as report_agent_module` sites** cannot be fixed by the harness edit alone (they bypass `make_report_agent` or use their own local import), so each is edited individually â€” five files total, enumerated in proposal.md and tasks.md.
5. **`deepear/src/agents/report/__init__.py` re-exports by explicit name, not `from .agent import *`.** A wildcard import would silently omit every underscore-prefixed name (none are re-exported here, all leaf-module public functions are already underscore-free) but more importantly would make the re-export list implicit and unreviewable; explicit `from .module import name` lines match every other package `__init__.py`'s style in this repo and make `__all__` a simple restatement, not a derived guess.
6. **The shim re-exports only `ReportAgent`.** The pre-move name inventory of `deepear.src.agents.report_agent` (`dir()`) contains 45 names â€” but 44 of them (`Agent`, `Model`, `DatabaseManager`, `InMemoryRAG`, `StockTools`, `ForecastAgent`, `ClusterContext`, `ForecastResult`, typing aliases, `re`/`time`/`datetime`/`timedelta`/`SimpleNamespace`, `logger`, every `_*_impl` alias, every `deepear.src.prompts.report_agent` function) are incidental to `report_agent.py`'s own imports, not part of its public surface, and the audit (proposal.md) found zero consumers or tests referencing any of them via the `report_agent` module path except the three (`Agent`, `ForecastAgent`, `_cluster_signals_impl`) being explicitly repointed to the new namespace by this same change. Re-exporting them from the shim would not restore old patch behavior anyway (Decision 1) and would misleadingly suggest the shim is a live seam for names it cannot actually intercept. Documented in the shim's own docstring so a future reader does not assume otherwise.

## Risks / Trade-offs

- This is the lowest-risk step in the program in one sense (moves a class whose every collaborator was already extracted and characterized by seven prior steps) but the highest-risk in another (it is the first Phase 4 step where a real, load-bearing patch seam actually needs test edits, not just a re-export) â€” the risk is entirely in getting the enumeration complete, closed by the `git grep` audit in proposal.md and tasks.md plus the experimental before/after check in tasks.md Â§3.
- Leaving `deepear/src/agents/__init__.py`/`deepear/src/main_flow.py`/every test file importing `ReportAgent` from the old `deepear.src.agents.report_agent` path (rather than migrating them to `deepear.src.agents.report.agent` or `deepear.src.agents.report`) is intentional per the plan's deferred step 32 â€” changing them is not required for correctness and would inflate this change's diff without reducing risk.
